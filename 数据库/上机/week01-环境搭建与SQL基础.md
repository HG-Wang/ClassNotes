# 实验一：环境搭建、关系模式设计与 SQL 基础查询

## 一、实验目标

1. 搭好 PostgreSQL 数据库环境，能连上、能用
2. 理解关系模型的核心概念：表、主键、外键、参照完整性
3. 学会用 DDL 建表、加载数据
4. 写出基础的 SELECT / WHERE / ORDER BY 查询

## 二、相关知识点

- **教材对应**：《数据库系统概念》第七版 第1-3章
- 关系模型基础：域、属性、元组、关系
- 主键约束与外键参照完整性
- SQL 数据定义语言（DDL）与数据操纵语言（DML）
- 基本查询：SELECT / FROM / WHERE / ORDER BY / LIMIT

## 三、数据来源

**豆瓣电影数据集**（约 3 万部电影）

- 来源：Kaggle 公开数据集或教师预处理版本
- 包含 5 张表：
  | 表名 | 内容 | 主要字段 |
  |------|------|---------|
  | `movies` | 电影信息 | movie_id, title, year, rating, votes, genres, duration |
  | `directors` | 导演信息 | director_id, name, birth_year |
  | `movie_directors` | 电影-导演关联 | movie_id, director_id |
  | `actors` | 演员信息 | actor_id, name |
  | `movie_actors` | 电影-演员关联 | movie_id, actor_id, role |
- 格式：CSV 文件，老师会发下载链接

> 老师会提供 CSV 文件包和建表脚本的**半成品模板**（只有框架，关键约束留空），你需要自己补全。

## 四、环境准备

详见《环境配置指南》。本次实验需要：
- PostgreSQL 数据库（Docker 或本地安装）
- DBeaver 或其他客户端工具

## 五、实验要求

- 可以用 Kimi、通义千问、GLM 等查语法、理解概念
- **建表语句必须自己写**——不能让 AI 直接生成完整 DDL 然后复制粘贴
- 每一步都要在 PostgreSQL 里实际跑过，截图留证

## 六、实验步骤

### 阶段 A：把环境跑起来（约 25 分钟）

1. 按《环境配置指南》启动 PostgreSQL
2. 用 DBeaver（或 psql）连上数据库，执行 `SELECT version();` 确认连接正常
3. **检查点 1**：截图——连接成功界面 + 版本号

### 阶段 B：建表、导数据（约 35 分钟）

4. 看一下 CSV 文件的表头，然后**自己写** CREATE TABLE 语句：
   - `movies` 表：主键 `movie_id`，注意选合适的类型（比如 `year` 用 SMALLINT，`rating` 用 NUMERIC(3,1)）
   - `directors` 和 `actors`：各有主键
   - `movie_directors` 和 `movie_actors`：复合主键 + 外键约束
5. 用 `\copy` 或 DBeaver 导入功能把 CSV 数据灌进去
6. 验证数据是不是对的：
   ```sql
   -- 看看各表有多少行
   SELECT COUNT(*) FROM movies;
   SELECT COUNT(*) FROM movie_directors;

   -- 试着插一条违反外键的数据，看会不会报错
   INSERT INTO movie_directors VALUES (999999, 1);  -- 这条应该失败
   ```
7. **检查点 2**：截图——五张表的行数 + 外键报错信息

### 阶段 C：写几条查询练练手（约 30 分钟）

每道题记录你的 SQL 和执行结果：

8. **Q1**：找出评分最高的 10 部电影，显示片名、年份和评分
9. **Q2**：找出 2020 年之后上映、评分 > 8.0、投票数 > 10000 的电影
10. **Q3**：按年代统计电影数量（1990s、2000s、2010s……），从多到少排
    - 提示：`FLOOR(year / 10) * 10` 可以算出年代
11. **Q4**：找出片名里带"爱"字的电影，按评分从高到低
12. **检查点 3**：截图——4 道题的 SQL + 结果

### 实验收尾：学习反思（最后 5 分钟）

13. 用**自己的话**回答：
    - 为什么要设外键？如果不加外键约束，数据可能出什么问题？
    - `movie_actors` 这种关联表是干嘛用的？能不能直接在 `movies` 表里加个字段存演员名字？为什么？
